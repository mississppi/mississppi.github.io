#　 LLM の整理

ニューラルネットワーク

ニューロンを模倣した計算モデル 情報を層で少しずつ変換して処理する。という特徴 段階的に抽象度を上げながら理解を深めていく。
LLM は何百層にもなっていて、複雑なパターンが読み取れる

LLM の進化

2 つの強みで進化してきた。

1. 事前学習　 Pre-training

一般的なルールやパターン。百科事典的な

2.　ファインチューニング

特定の目的に合わせて微調整する。

LLM の学習について

自己教師あり学習という方法。ラベルがない。テキストとのものでモデル訓練している。 また、このモデルをスケールさせればタスク性能も向上するという仮説で GPT-3 くらいまではきていた。

自然言語処理について

言葉をコンピュータに理解させる技術のことを指す。これは昔は専用のモデルが必要とされていた。 近年はそうでもない。汎用 LLM とも言われてきている。

汎用 LLM について

得意なこと

・テキスト生成 ・質問応答 ・翻訳・ようやく ・コード生成

機械学習との違い

そもそも、「データからルールを自動で学習する」技術。ただデメリット ・特定のタスクに対して特定のモデル。 ・学習には大量ラベル月データが必要 ・新規タスクは、ゼロからモデル構築

LLM は全く異なるスタンスとなる。 ・事前学習で得ている言語の一般知識 ・トランファーラーニングによるタスク適応 ・スケーラブルなモデル

トランスフォーマ

LLM は、従来のモデルである RNN, CNN とは別のアテンション概念を使っている。 Tranformer, Transfor ラーニングで構築。これが根本から違うらし。

トランスフォーマ

RNN, LSTM だと長文が文脈保持が難しかったが、並列処理が可能で、単語間の関係も効率的に捉えることができる用になった。

主要構造

エンコーダ(BERT が特化)、でこーだ(GPT)、T5(両方)

##　最大の特徴 self-attention-mechanism と呼ばれる、各単語の関連を計算する仕組み。 とにかくこれが文脈を深く理解し、適切なテキスト生成のために欠かせない仕組みとなっている。 具体的には、入力テキストの各単語が、文中の他の単語とどの程度関連してるか、数値的に計算する。重みを調整することができる。

Attention の仕組み

スケールドットプロダクトアテンションと呼ばれる処理について設名する。

Query: 注目する単語 Key: 他単語の情報、比較の対象 Value: 関連性に基づいて出力される意味情報

ここからが本題。処理の手順

  1. クエリとキーの内積計算
  2. ソフトマックス関数を使う。スコアを確率分布に変換
  3. vaalue を重み付で加算。ベクトルを出力

さらにマルチヘッドアテンションを使い強化している。 これは以下となる

  1. 複数の自己注意メカニズムをヘッドを並列に実行 それぞれのクエリキーバリュー行列を用いる

  2. それぞれのヘッドを結合、結合ベクトルを生成

  3. 最終的に、統合ベクトルを活用して文章を処理

これで単語間の関係を複数の視点から同時に分析できる。高度なことができる。

LLM のトレーニング方法

どうやって言葉を理解するのか。 インプット: ニュース、書籍、web、会話、コードなど.　　 データセットとも言われる。

ただしこれらはクリーンな状態のデータにする必要がある ・スパムの除去 ・html タグの除去 ・絵文字 ・重複データの削除 ・文法ミス、誤字修正

揺れも問題になる。Normalization する。

tokenization と呼ばれるが、単語分割する。 ・スペースごとに区切る ・形態素解析。品詞単位 ・サブワード。よく出現する部分単位で分割

ただし、バイアスかかるため偏ったデータは好ましくない

トレーニングステップの概要

まず膨大なデータと計算資源がいる。具体的には次のステップ

  1. 初期化-モデルのパラメータをランダムに設定する
    ランダム初期化、Xaview/Glorot 初期化、He 初期化などがある

  2. フォワードプロパゲーション-入力を処理し、予測結果を生成 ここの目的は入力された文章を適切な予測を作成。

2-1. 入力データの埋め込み(Embedding): テキストをベクトル化 2-2. トランスフォーマ層での処理: self-attention で単語間の関係性を考慮する。FFN によって情報を変換 2-3. 最終出力の生成: LLM は次の単語を予測するよう出力(GPT)。文章全体の文脈理解で分類タスク(BERT 系)

  1. ロス計算-予測結果と正解データの誤差を測定する 誤差測定、最小化にしたい。 LLM ではクロスエントロピー損失関数が利用される。 これは何か。

    ロス計算したら、誤差を下にパラメータ修正をするバックプロ派ゲーション(逆電版)を行う。これで正確な予測が可能になる

  2. バックプロ派ゲーション-誤差を下にパラメータ修正する 勾配降下法: ロスを小さくする方向にパラメータを更新する 最適化手法: パラメータ更新をスムーズに行うアルゴリズム(Adam, RMSProp, SGD)

  3. エポックの繰り返し-上記プロセスを何百万回も繰り返す。モデル改善 トレーニングの回数はエポックとも呼ばれる。過学習のリスクもあり

トレーニングの際的あ k は、学習率とハイパーパラメータがカギ。

4-1. テキスト生成

LLM のテキスト生成は、次に続く単語、フレーズの予測に基づいてる。 GPT は、文の前方の情報に基づいて自然の文章の続きを生成する単方向。各単語がどのように前後の単語と関係しているか計算して、結果を下に最適な次の単語を選んでいる。 自己回帰型モデルは、生成された単語も入力となる。、さらに新しい単語を生成するというプロセスを繰り返す。なので一貫性のある文章となる。

利用される技術は以下。 ・トークナイゼーション: 単語やサブワード単位に分割して、モデルが処理しやすい形式に変換する。 ・確率分布による単語選択: 確率の高い単語を選ぶ(Greedy Search, TemperatureSampling, Top-kSampling) ・フィルタリングと制御: 事前に禁止語を設定する、トーンを調整する。インプットを工夫して品質方向性をコントロール

品質とリスク

・バイアス(ジェンダー、文化、政治) ・誤情報の生成: 事実とフィクションを区別できない。 ・倫理、法的問題: 著作権、不適切コンテンツを生む。

今後について

・パーソナライズ: 好み、過去の対話履歴を学習。個別に最適化された文章 ・対話型 AI の進化。意図や感情を理解し応答 ・リアルタイム生成の精度: カスタマーサポートはよりかつようされていく

4-2. 質問応答システム

この仕組みとして、テキスト文脈を理解し、質問に対して最も適切な回答を抽出する、という仕組み。
BERT では、双方向の文脈理解で、全体の意味を捉えて質問に答えることができる。単語間の関係性を深く分析、意味を文脈に照らして理解することができる。 学習済みモデルを利用すれば、膨大なデータから一般的な知識を吸収し、幅広い分野の質問に対応できる。

・よくでてくる記号になれよう

記号式意味
P(x)x が起きる確率。例）P(”猫”)=0.04 は、次の単語が猫になる確率が 4%
P(AB)
Σ足し算の繰り返し（総和）。例）ΣP(w) は、すべての単語 w の確率を足し合わせて１になるべき。てきな。
E[x]期待値: x の平均的な値. 例) E[単語の長さ]=5.2 は、ある文章で平均的な単語の長さが 5.2 文字
log(x)情報量を測る時に使う関数 x が小さいほど大きなあたいになる。れい）log(P(x)) x の置きにくさ=情報の驚度、p=0.01 なら log(P(x))は大きい負の数になる

たとえば。モデルの本質の式

P(w_t|w_1,w_2,...,w{t-1})

これはこれまでにでてきた単語 w_1 から w_1{t-1}を踏まえて次に w_t がくる確率はどれくらいか。という本質。 他にも ・文章で次にくる単語の候補をどう評価するか ・情報量が多い単語はどれか ・モデルがどのくらい迷っているか を数式で表現するのが LLM

確率論。単語はどう選ばれる？

LLM は文章を予測の連続として処理する。

P(次の文脈|これまでの文脈) P(“晴れ”|”今日は天気が”) = 0.62 P(“悪い”|”今日は天気が”) = 0.18 P(“謎だ”|”今日は天気が”) = 0.01

  1. ユーザが入力
  2. モデルが予測
  3. 単語が選択される。 これが繰り返される。自己回帰的生成(autoregressive generation)と呼ばれる

ただランダム性もあり、Top-K サンプリング、Temperature 調整といった揺らぎ調整も持たせている

迷いについて

LLM において、単語予測でモデルが迷いの状態になることがある。これの度合いをエントロピーと指す。 情報量から考える

I(x) = -¥log2P(x)

珍しい出来事ほど多くの応報を持つ

エントロピーとは

「ある状況において、平均してどれくらい情報が必要か」を表す量。

H(X) = -ΣP(x)¥log2P(x)

早い話予測の難しさ。enum が 3 つあるのか、x つあるのか。不確実生の度合いを図る物。 不確実な情報を、ビットという単位でどれくらい表現できるか

クロスエントロピー

H(p,q) = -Σp(x)¥log2q(x)

正解ラベルが正しく予測されていたかどうかを評価するもの。モデル学習ではクロエンを損失関数として利用。 正解に近づけるよう調整してる

RAG。なぜ今なのか。

ChatGPT の限界.4 つの視点

  1. 自社固有情報を持ってない構造的制約
  2. ハルシネーション
  3. 最新情報非対応
  4. 再学習、カスタムが無理

ここで RAG がでてくる、「生成前に情報を渡す」という発想

モデルに詰め込みの限界

  1. コスト高い
  2. メンテ負荷高い
  3. セキュリティ、プライバシー

RAG のフロー

  1. ユーザの入力受け取る
  2. ベクトル検索エンジンが、入力に関するドキュメント、ナレッジを社内けんさく
  3. 検索結果をコンテキストとして LLM に渡す。回答を生成する。

ナレッジの外部化のメリット

explainability の説明可能性を担保できる。この文書に基づいて回答しました。という出典情報を添えることが可能となっている。

現在は以下となっている。 ・モデルは知識を保持しない。参照する ・生成の品質は、渡された知識の構造と文脈設計に依存する。 渡す情報とタイミングの調整が、設計の力点が置かれる用になっている。

RAG の始まり

・2020 年に Meta が発表 ・NeurIPS で発表。パトリックルイスの論文で発表してから広まった ・それまでは色々でてきていた。

2013: word2vec 2018: BERT 文脈加味した双方向モデル 2019: GPT-2/BART 2020: DPR せまんてぃっく検索ができるようになった 2020: RAG 検索と生成統合ができた。

基本フロー

  1. 入力受付: 質問受付
  2. 検索: ベクトル変換し意味の違い文書を検索。これは Retriever が担当
  3. 統合: 検索結果をプロンプトに組み込む
  4. 生成: LLM が文脈に基づき回答を生成する。Generator が自然言語の応答を生成

Retriever, Generator の分離モデル、役割

Retriever の役割:関連情報を探す

役割: 質問に関連する情報をナレッジベースから検索し LLM に提供する。 意味ベース検索(セマンティック検索)で実行、キーワード一致ではない。

・埋め込みベクトルに変換 ・ナレッジ内の文書ベクトルと比較 ・類似度の高い n 件返す

ここ検索品質が低いと、その後にも影響する

Generator の役割: 自然な文で応答する

Retriever からの情報を文脈として、LLM に渡して自然な応答を生成する。 ・プロンプト設計によって情報を LLM に組み込む ・質問、検索文書を統合し、応答を作成 ・要約や言い換えで自然文を出力

ここがすごく設計で重要

分離設計のメリット

  1. 知識の更新がナレッジ側のみ更新で oK
  2. 性能最適化。各要素を個別に最適化 (検索制度 vs 応答品質)
  3. 責任分担
  4. モジュール設計としてできる。

従来検索と RAG の違い

これまでは、キーワード一致する文書を単純に探す。関連スコアの高い順に表示する構造。

・キーワード一致に依存(“返品ポリシー”など) ・表現の揺らぎに弱い ・応答ではなく情報提供型 ・文脈を考慮しない

RAG の検索は以下となる。 ・Retriever では、「退職者支援制度」「健康保険の継続」といった意味的に関連する文書を取得する

RAG の得意不得意

得意なこと ・ナレッジ検索・FAQ の高度化: 明文化された社内ナレッジ、FAQ ベースにした応答 ・専門文書を根拠とした応答 ・非構造データからの知識抽出: 議事録、チャットログ、問い合わせ履歴から検索、生成できる点も RAG の利点。 これは Retriever が文脈単位のチャンクを抽出でき、Gene が一貫した回答を掲するので知識が再利用できる

不得意なこと ・数値処理。論理推論が必要な問い: 例) 昨年の売り上げ前年比を計算して。計算処理、ルールエンジンがいる。 ・リアルタイム性の高い情報提供: 株価、てんき、交通。API 連携がいる ・会話文脈の継続、状態管理が必要な対話: 複数ターンの文脈保持、記憶するユースケースには MCP が推奨される

判断軸

・回答の根拠が文書として存在しているか。 ・知識は静的か。毎回更新の必要はないか ・問いの表現が多様でテンプレ対応できないか。 ・専門性高い。LLM 単体では不足か。

必要な技術コンポーネント

実際は複数の技術を組み合わせる必要がある。とりあえず 4 つ。

4-1. 埋め込みモデル embedding model

中核の技術。自然言語 to 数値ベクトルにする技術。 これがあると Retriever の意味ベース検索ができる。 埋め込み: 意味を機械が検索可能な形式で表現すること。これで「意味が違いかどうか」判断できる。

なんで必要か。

・言い換えに弱い。検索漏れ ・同義語、略語に対応できない ・意味の近さを捉えられない

これらは質問と文書ちゃんくを元にベクトル化する。意味の近さ(コサイン類似度)で照合する。 このベクトル化するのが埋め込みモデル。意味空間における近接位置のマッピングで可能となる。

代表的なモデル

  1. OpenAI Embedding (text-embedding-ada-002) ・OpenAI API で使える ・1536 次元 ・他言語 ・ChatGPT と統合性高い。精度と使いやすさのバランス ・外部 API なのでプライバシー管理、レイテンシに配慮

  2. SBERT(Sentence-BERT) ・OSS でローカルでも動作 ・384, 768 次元 ・cl-tohoku など日本語もあるっちゃある ・オンプレや機密データあってもいい ・精度はモデル選定、チューニングに依存

  3. Cohere, Google USE, Instructor XL ・各社の Embedding モデル。API で使える ・Cohere など zero-shot, MultiLingual 対応に強い ・RAG 向けに設計された汎用埋め込みモデル増加中

選定時のポイント

精度: 質問と対応度がどれだけ高いか 日本語: 混在次の品質 実行環境: クラウドかローカルか トークン数制限: 長文処理の対応力。文チャンクの長さ メンテ性: モデル更新頻度。継続利用の安定性

実施は文書のチャンク化(分割単位の設計)がめちゃ大事。 ・短いと文脈が失われる ・長いと Embedding が曖昧になる。意味が散る

最適なチャンクは、1 つの意味単位が保たれつつ、モデル最大トークン数に収まる程度の分割。 チャンクには、メタ情報(カテゴリ、更新日、文書名など)を付加することで結果のフィルタリングや、優先順位制御に対応できる。

4-2. ベクトル検索エンジン faiss weaviate pinecone 埋め込みベクトルだけだと意味的に類似する文書を効率よく見つけられない。 ベクトル検索エンジンが必要。Vector Search Engine。これは数万、数百万の高次元ベクトルからユーザの質問に近い情報を高速かつ正確に検索するためのインフラとなる。

なぜ必要か

距離関数をつかって類似性を検索しているから。

基本構造

インデックス: 埋め込みベクトルを格納、構造化。高速検索できるようにする。圧縮・近侍・シャーディングが使われる。 検索アルゴリズム: 対象となるベクトルに対して、指定した類似度関数(コサイン類似度、ユークリッド距離)を使う。上位 N 件。 めたでーたけんさく/フィルタ機能: カテゴリやタグ、日時などの付加情報に基づく条件付き検索も可能(ハイブリッド検索) スケーラビリティ: 数百万規模のベクトル、高頻度アクセスをささえる分散処理・メモリ設計。

主なエンジン

  1. FAISS(Facebook AI Similarity Search) 大規模処理に強い。GPU 対応。ローカルに最適 管理画面、REST がない。統合に手間

  2. Weaviate OSS。REST, GraphQL ベースの操作。クラウド、オンプレいける メタ検索、スキーマ設計・ハイブリッド検索もできる。langchain も相性がいい。 日本語モデルが怪しい

  3. Pinecone 商用。ベクトル DB。スケールする、安定、API 使いやすい。ハイブリッドもできる 故にお金がかかる。

選定時はスケールするか、運用携帯、フィルタ機能、言語、他ツール連携などがあるかを勘案する 検索結果が過剰になったり、関係ないものも検索される場合 g あある。 ・スコア設定 ・ラグランク ・正規化する

あくまで意味空間における候補抽出の第一歩となる。

4-3. コンテキスト整形 - llm 統合

とにかく整形が必要

Retriever によって取得された文書は、通常は複数の非構造のチャンクとなる。 このままモデルに投げても精度がでない。

どの情報を、どの順で、どのような文脈で提示するかを設計しないといけない

基本的な整形フロー

  1. チャンクのスコアづけと並び替え 類似度やヒット率の高い順に並び替える。重要後やメタ情報を加味して再評価することもある。

  2. チャンクの選定(トークン最適化) トークン上限(4096/8192/32k)を意識して、必要な文書のみを選択。

  3. 整形テンプレートへの組み込み ただ連結するのではなく構造的なフォーマットに整え LLM に渡す。

整形時のポイントとしては、

・同種の情報は近接させる(話題の散乱を防ぐ) ・重要な情報を先に配置する ・出典や文書タイトルを明記する ・文書の間には明確な区切りを儲ける(コンテキスト境界を誤解させない)

ここをプロンプトエンジニアリングと言われてる

4-4. セマンティック検索とキーワード検索の違い

keyword 検索とは

ElasticSearch, Solr など TF-IDF、BM25 などが挙げられる。 早い。精度よし、安定。 言い換えには対応できない

セマンティック検索

意味が近ければヒットする。 関連情報を抽出しやすい。 単語の一致に頼らない。漏れを減らせる 薄い文書が引っかかるリスク 実装に Embedding, ベクトル検索エンジンがいる

ハイブリッドが注目されている

どっちでも検索する。 ・検索段階で両方走らせ、結果を合成(スコア加重) ・キーワードフィルタ、意味スコアでランキング ・セマンティック検索後にキーワード検索による再ランクを実施

Weaviate, Pinecone などはハイブリッドが標準みたい。
