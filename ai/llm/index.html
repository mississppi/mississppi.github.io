LLM を理解する

基本的な設名としては、文字列を入力すると文字列を返すサービス。 入力: プロンプト 出力: 補完、レスポンス

トレーニングされていない LLM は何にも使えない。のでトレーニングする必要があるが、 普通は済みの基盤モデルを使って進めていく。 ファインチューニングを加えることもある。

ファインチューニングとは

OpenAICodex の初期バージョンでは、GPT-3 をベースで、Github のソースコードでファインチューニングされたという話がある。

LLM は大量のデータセットという大量のドキュメントの集合でトレーニングされる。 目的に依存するので色々なものが食わされる。モデルはこのデータセットからトレーニングデータセットに似た出漁を生成する方法を学習する 例えば、トレーニングデータセットに含まれるドキュメントの冒頭部分をプロンプトとして与えると、モデルは自然に通 d きそうなテキストを補完する。 要は模倣しようとする。

巨大エンジンと何が違うか。LLM の目標は丸暗記ではない。再現ではない。 ・その中で見出したパターンと、論理的な推論展開の仕方を抽象化し、データとは異なる新たなプロンプトに対しても補完できるようになること ・丸暗記は失敗となる

ドキュメントの補完

・トレーニングデータをよく知れば、トレーニングされた LLM の出力も直感で形成しやすい

人との思考の違い

LLM は統計的に最もありそうな補完を選ぶ。これは人間が行う前提とは異なる場合がある。 人間はテキスト作成のプロセスでもっと広い文脈、意図が関与するから。 モデルは常に推測する、パターン模倣するので実在しそうな情報をもっともらしく紹介する

ハル氏ネーション

幻覚。もっともらしく間違う。 指摘しても効果はない。 ・検証可能な背景情報を提供させるアプローチが一般的 ・参照リンクなど 誘発されることもある。実在しないものに言及すると、LLM は本当に存在すると想定し続ける。 これが真実バイアスと呼ばれる。

##LLM の見る世界

LLM も一文字ずつ読むわけではない。 トークン単位でチャンク分解される。トークンの集合を語彙と呼ぶ モデルはまずトークナイザーに通して、トークン列を変換する。この列を LLM に送る。 LLM は、列(内部は数値)を生成してトークナイザーを経由してレスポンされる — OpenAI のトークナイザー

LLM はトークン集合をみてる、人間は単語の集合をみてる

###違い１：　 LLM は決定論的なトークナイザーを利用する 単語への変換を曖昧にしてる人間に対し、LLM は決定論的なトクナイを使う。 でも大抵誤字にもなれてるので強い

＃＃違い２：　文字をひとつずつゆっくり確認できない 人間は一文字ずつゆっくり認識できる。LLM はトクナイを使うので難しい。 トークンを分解して再構築は全然できないと認識する。

＃＃違い３：文字を人間とは異なる見方をする 大文字小文字で利用されるトークン数が変わってくる

トークン数を数える

モデルとトークナイザーは混在 NG。自分のモデルのトークナイザーを深く理解しておきたい。 とにかくトークン数で色々きまる。 上限がある。好きなだけ長いテキストをやりとりはできない。 文字数をトークン数に変換する公式は無い。平均 1 とーくん約４文字。 英語以外も効率落ちる。 数字だけとかになると効率落ちる。 平均 2 文字ちょっとで 1 トークンとか。 珍しい文字を含む場合も効率さらに落ちる。

1 つのトークンずつ

実際はテキストからテキスト変換をしてるわけではなく、トークンとーーくんへの変換もしてない。 実際は複数トークンから、次の一つのトークンを求め続けているこれを繰り返し単一トークンを積み重ねている

自己回帰モデル

LLM に処理をさせると、統計的に最も高い次のトークンが得られる。 これが末尾に追加される。今度はこれを新プロンプトとしてまた次のトークンを要求する。 これを自己回帰と呼ばれる。 トークンは途中で覆せない。これは完成前のドキュメントでトレーニングされていないから。 時には頑固に見える。軌道修正はアプリ設計者がやらないといけない。

問題点：モデル自信が作ったパターンにハマること。パターン認識が得意なので、パターンが見つかるとやめられなくなることがある。

temperature と確率　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　

実際にはモデルはあり得る全てのトークンの確率を計算する、そして 1 つを選び出している。 これはサンプリングと呼ばれる。 確率を計算して確率情報を返すことができる。これを logprob という形式で返す。 これが高いほどより有力となる。だいたい-2 から 0 の間くらい

・logprobs 3 にすると、3 つの候補が取得できる。これを選ぶという戦略も取れる。

0: 代替が不要で優良なもののみが欲しい 0.1 - 0.4 : ごくわずかに劣るトークンをたまにでも拾って欲しい。 0.5 - 0.7: 偶然性の影響を大きくしたい場合。多様性を重視する時 1: トレーニングエータの確率分布を反映させる。51%, 31%となったらその通りになる。

1: 乱雑なものが欲しい。

tempe が高い: 多くの代替案 tempe が低い: より正確な回答

サンプリングの代表でビームサーチがある。複数先のトークンまで視野にいれて連続性を確保。でも多大なコスト

トランスフォーマアーキテクチャ

LLM の頭脳は 1 つの脳ではなく、小さい脳が無数にある。構造は同じ。各トークンに 1 つずつミニが割り当てられてる。 これをトランスフォーマと呼ぶ

アテンション: ミニブレインどうしが情報を受け渡す手段。数千ものミニが互いに情報を共通している可能性もある。

受け渡しの流れは、回答前の左側にしか聞けない。

チャット形式への移行

基盤モデルは基本使いにくい。これを人間のフィードバックの RHLF できると。強化学習。 OpenAI, davinchi-002 にインターネット情報を全て食わせたが回答が不安定。 ここでモデルのアラインメントが必要になる。 kokode ChatGPT で普及したモデル一覧

・教師ありファインチューニングモデル: 指示にしたがってチャットする。 ベースモデルに数万ん件規模の対話データでトレーニングされたもの。調整用という認識

・報酬モデル: 補完の品質をスコアリング 強化学習。とりあえずランキングのデータを用意する。良い悪いの

・RLHF モデル: チャットし有用、正直、無害を満たす どうやら RLHF モデルを進んでも、スコア向上のための抜け道を作ろうとする。ので、 PPO というアルゴリズムを使う。

こうしてようやく、「内部知識と食い違う回答はよくない」「内部知識に合致する回答は良い」という規則を学習する ようやく元の情報源を参照してください、的なコメントをするようになる

アラインメント税に注意。

RLHF のプロセスによってモデルの知能が失われる場合もある。OpenAI はこの辺りうまく調整している

どんなドキュメント作ればいいか

  1. チャットぽい
  • 自然にできる
  • テクニックとして、構造化にして回答の最初を記載してしまう。これでモデルが回答せざるを得ないことになる。
  1. 分析レポートっぽい
  • markdown が良き
  • CoT プロんプティングてなにゃねん

  • 結論の前に #アイデア、分析を追加することで、いい感じになる

  • 結論の前に #参考文献を追加することで、レスポンスをいつ終了するべきか、示すことができる。これを停止シーケンスとするテクニックがあるらしい
  1. 構造化ドキュメントっぽい
  • anthropic artifacts プロンプトがある。共同編集自己完結型ドキュメント。
  • 右側に UI が表示されてるものである
  • xml,yaml どっちも使えるよ。

・とにかくコンテンツをスニペットに変換しなくては行けない。 テクニックがある。
